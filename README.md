## 🧩 한 줄 소개

기존 보이스피싱 방지 솔루션은 대부분 **사후 신고된 발신번호 차단** 중심입니다.
즉 통화가 끝난 뒤에야 신고하거나, 이미 알려진 사기 번호만 걸러냅니다.

하지만 실제 피해는 **통화가 진행되는 그 몇 분 안에** 발생합니다.
SSAIREN은 통화 도중 음성을 실시간으로 분석해 위험을 감지하고, 사용자가 직접 입력하지 않아도
**AI가 가족과 경찰에게 동시에 대응**하도록 설계했습니다.

## 🏗️ 시스템 아키텍처

```
 Flutter ──REST(평시) / WebSocket(위험 시)──▶ Spring ──REST──▶ FastAPI
   │  온디바이스 Whisper STT                  │ 1차 필터링        │ 2차 정밀 분석
   │  (5초 청크 → 텍스트만 전송)               │ 게이트키퍼         │ LangGraph Agent
   ▼                                         ├── PostgreSQL      │
 FCM 푸시 알림 ◀────────────────────────────┘                   └── location / family / police 도구 실행
```

- **Flutter 앱** — 온디바이스 경량 STT(Whisper)로 통화 음성을 5초 단위 텍스트로 변환해 전송 (음성 파일이 아닌 **텍스트만** 보내 보안·경량성 확보)
- **Spring (게이트키퍼)** — 클라이언트 facing 담당. 키워드·룰 기반 **1차 빠른 필터링**, DB, FCM
- **FastAPI (AI)** — LangGraph 기반 멀티 에이전트로 **2차 정밀 분석** 및 대응 도구 실행
- **인프라** — EC2 + Docker 배포, 관제 대시보드는 React(Vercel)

### 2-Layer 위험 판단 구조 (서비스 핵심)

| 단계 | 담당 | 내용 |
| --- | --- | --- |
| 1차 | Spring | 키워드·패턴 기반 빠른 스코어링. 명백히 안전한 대화는 여기서 걸러 비용·지연 최소화 |
| 2차 | FastAPI / LangGraph | `situation detector`로 시나리오 분류(납치협박·기관사칭·가족사칭·계좌이체) → `scenario analyzer`가 긴급성·금전요구·고립정도·신원위장·행동패턴 **5축**으로 위험도 산출 |

**위험도 구간**

| 구간 | 점수 |
| --- | --- |
| 안전 | 0 ~ 40 |
| 의심 | 41 ~ 55 |
| 경고 | 56 ~ 75 |
| 위험 | 76 이상 |

위험 단계로 격상되면 LangGraph가 `location_check()`, `send_family_alert()`,
`notify_police()` 등 대응 도구를 **사용자 개입 없이 직접 실행**합니다.

### 기술 스택

`Flutter` · `Spring Boot` · `FastAPI` · `LangChain / LangGraph` · `Whisper STT` ·
`KoELECTRA` · `PostgreSQL` · `FCM / Firebase` · `React` · `EC2` · `Docker`

## 📝 회고록

### 아이디어 구체화 — "되는 것"부터 확인하고 설계를 거꾸로

처음엔 *보이스피싱 탐지* 라는 방향만 있었고, 어떻게 구현할지는 막막했습니다.
"클로바노트의 실시간 음성 인식이 어떻게 되지?" 같은 질문에서 출발해
동시성·오디오 캡처·온디바이스 STT까지 기술 스택을 하나씩 검증해 나갔습니다.

macOS에서 BlackHole로 Discord 통화 오디오를 음성으로 분리해 STT를 돌리는 PoC를
직접 해보고, 정상 동작을 먼저 확인한 뒤 설계를 역으로 잡았습니다.
기술적으로 **"되는 것"을 먼저 확인하고 들어간 게** 이후 일정을 지탱했습니다.

### 트러블슈팅 — 막힌 지점들

- **Android 통화 오디오 캡처 불가**
  Android는 OS 정책상 통화 오디오 직접 캡처가 막혀 있습니다.
  *마이크 입력으로 상대 음성을 받아 STT* 하는 방향으로 선회했고,
  화자 분리는 데모 목적에 불필요하다고 결론 내렸습니다.

- **Flutter 빌드 실패 — `record_linux`**
  `record_linux 0.7.2`가 `record_platform_interface 1.6.0`의 `startStream`을
  구현하지 않아 빌드가 막혔습니다.
  `record` 패키지 전체를 올리면 breaking change로 오디오 코드를 다 고쳐야 하므로,
  `pubspec.yaml`에 `dependency_overrides: record_linux: ^1.3.0`만 추가하는
  최소 침습 방식으로 해결했습니다.

- **API 구조 단순화**
  WebSocket을 전 구간에 쓰려다, Whisper 레이턴시로 인한 청크 겹침 문제를 짚고
  **평시 5초 REST 폴링 + 위험 감지 시에만 WebSocket 전환** 구조로 단순화했습니다.
  폴링도 "이전 요청 완료 후 다음 요청 시작" 루프로 바꿔 중복을 막았습니다.

- **Vercel PATCH 403**
  로컬(내부망)에서는 되던 PATCH가 Vercel(외부 IP)에서 403.
  백엔드가 GET은 외부 허용, 쓰기 메서드는 내부망 IP만 허용하도록 설정돼 있던 게 원인이었습니다.

### 의사결정

- **트리거 방식** — 사용자 재심(PENDING + 승인) vs 자동 실행을 두고 고민하다,
  타겟이 노인이라는 점과 개발 복잡도를 이유로 **자동 실행**으로 정했습니다.
  중복 트리거는 `is_triggered` 인서트 플래그로 방지.
- **FCM** — 보호자 측 백그라운드 알림을 위해 필수임을 확인하고 포트폴리오에 포함.
- **Spring + FastAPI 역할 분리** — Spring은 게이트키퍼, FastAPI는 AI 추론 전담으로
  역할을 분리하고 내부 REST로 연결.

### PM으로서 — 코드를 직접 짜기보다 결정을 내리는 역할

아이디어 구체화부터 본선까지, 제가 한 일의 본질은 코드를 직접 짜는 게 아니라
**기술 타당성 검증 · 아키텍처 확정 · 역할 분담 · 리스크 관리**였습니다.
AI는 머릿속에 흐릿하게 가진 방향을 구체적인 실행으로 바꾸는 도구였고,
저는 그 결과물을 팀에 전달하는 역할을 했습니다.

산 건 설계 단계에서 충분히 검토하고 들어간 것이고, 아쉬운 건 팀원들과의 실제 싱크
과정이 회고에 충분히 남지 않아 제가 결정한 것들이 실제로 어떻게 반영됐는지를
다 보지 못했다는 점입니다. 결과가 어떻든, **PM으로서 AI를 쓰는 방식 하나는
이번에 제대로 익혔습니다.**
