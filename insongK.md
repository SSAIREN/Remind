## 1. 사람은 git 브랜치가 아니다 — 정보 동기화 문제

이번에 제일 자주, 제일 강하게 느낀 거다.

대화도 하고 노션·README로 문서화도 했는데, **사람들이 각자 머릿속에 들고 있는 "최신 버전"이 다 달랐다.** git은 `pull` 한 번이면 모두가 같은 HEAD를 보는데, 사람은 그렇지가 않다. 내가 어제 결정한 걸 누군가는 그제 버전으로 기억하고, 문서는 또 그 사이 어딘가에 멈춰 있다. API 구조를 WebSocket 전구간 → 5초 REST 폴링으로 바꾼 것 같은 결정이 머릿속·카톡·노션·코드에 제각각 다른 시점으로 흩어져 있었다.

문서화가 부족했다기보다, **문서가 "지금 진짜 맞는 버전"이라는 보장이 없었다.** 그게 핵심이다. 사람의 기억과 문서는 자동으로 merge되지 않는다.

**다음엔 이렇게:**
- 문서를 많이 쓰는 게 아니라 **단일 진실 소스(Single Source of Truth)를 하나만** 정한다. "결정은 여기에만 적는다. 여기에 없으면 결정 안 된 거다."
- **결정 로그(Decision Log)**를 짧게 운영. `날짜 / 무엇을 / 왜 / 바뀐 이전 결정`. 길게 쓰지 말고 한 줄씩.
- 큰 변경은 말로 흘리지 말고 **"공지 핀"** 한 곳에. 카톡에서 한 결정은 30분 뒤 사라진다.
- 결국 사람을 git처럼 만들 순 없으니, **동기화 의식(짧은 데일리 싱크)** 으로 강제로 HEAD를 맞춘다.

## 2. UI는 잘 잡았는데, 뒤에 숨은 API 명세가 너무 많았다

와이어프레임·UI는 솔직히 잘 나왔다고 생각했다. 그런데 화면을 실제로 움직이려니까 **화면 뒤에 안 보이는 명세가 생각보다 어마어마하게 많았다.** 통화 세션 생성, transcript 청크 시퀀스, 위험도 응답, WebSocket 이벤트 종류(`transcript_chunk`, `analysis_result`, `SESSION_COMPLETE_ACK`...), 가족 응답, FCM 토큰 등록... 화면 한 장 뒤에 API가 대여섯 개씩 붙어 있었다.

UI를 그릴 때 우리는 "보이는 것"만 설계했지, **"이 화면이 살아 움직이려면 어떤 데이터가 오가야 하는가"** 는 한참 뒤에야 마주쳤다. 그래서 명세를 제대로 못 했고, eventType이 뭔지·enum이 뭔지 카톡으로 계속 물어보는 상황이 반복됐다.

### 그래서 명세는 언제 해야 하는가? (내가 계속 모르겠던 거)

이번에 깨달은 답은 이거다 — **명세는 "와이어프레임 직후, 코드 직전"에 한다.**

화면이 정해지면 각 화면마다 "이 화면에 들어올 때 / 버튼 누를 때 / 떠날 때" **어떤 요청이 나가고 어떤 응답이 오는가**를 한 줄씩 적는다. 이걸 **화면 단위로 역설계**하면 숨은 API가 그때 다 튀어나온다. UI 그릴 때 같이 "데이터 흐름"을 옆에 적었어야 했다.

핵심은: **명세는 별도 단계가 아니라 와이어프레임의 일부**라는 것. 화면만 그리고 끝내면, 숨은 절반을 코드 짜다가 발견하게 된다.

## 3. 백엔드 Swagger가 미쳤다 — "그냥 연결하니까 되는 느낌"

문성원·박민영이 짜준 Swagger 명세가 진짜 편했다. 프론트 입장에서 **그냥 보고 연결하니까 됐다.** 추측할 필요가 없고, 요청/응답 모양이 거기 다 있으니까. 이게 좋은 협업 인터페이스구나를 몸으로 느꼈다.

여기서 배운 것: **좋은 명세는 문서가 아니라 "계약(contract)"이다.** Swagger처럼 실행 가능하고, 서버와 자동으로 같이 움직이는 명세는 2번에서 말한 "정보 동기화 문제"를 구조적으로 없애준다. 코드가 곧 명세니까 버전이 어긋날 수가 없다.

> 다음엔 이걸 **프론트 쪽에서도 먼저 요구**하자. "화면 그렸으니 이 화면에 필요한 API를 Swagger에 먼저 박아주세요"라고. 받아쓰기만 한 게 아니라 내가 끌어냈어야 했다.

### ERD랑 백엔드는 언제 설계해야 했나

이번 순서를 돌아보면, ERD가 화면·명세보다 살짝 늦거나 뒤섞여 진행됐다. 내가 생각하는 이상적인 타이밍:

```
주제·기능 확정
   → 와이어프레임(화면)
   → 화면별 데이터 흐름 = API 명세 초안  ← 여기서 "필요한 데이터"가 드러남
   → ERD (그 데이터를 어디 저장할지)
   → 백엔드 Swagger 스켈레톤 = 계약 확정
   → 프론트·AI·백 병렬 개발
```

즉 **ERD는 "화면이 요구하는 데이터"가 보인 다음**에 짜는 게 맞다. 데이터 모델을 먼저 추상적으로 짜면 화면이랑 안 맞고, 화면만 짜면 저장 구조가 안 보인다. **API 명세가 둘을 잇는 다리**였다. 이번엔 그 다리를 늦게 놨다.

## 4. "그거 위험 뜨는 거 있잖아, 확인 누르면" — 지칭의 지옥

매번 느끼는데, 같은 걸 부르는 말이 사람마다 다 다르다. "그거 위험 뜨는 거", "그 빨간 거", "확인 누르면 나오는 거"... 나는 A 화면을 말하는데 듣는 사람은 B를 떠올린다. 코드엔 `harmful_bottom_sheet`, `risk_badge`, `RiskLevel.HIGH`처럼 정확한 이름이 있는데, **말할 때는 그 이름을 안 쓴다.**

이게 사소해 보여도 회의 시간의 절반을 잡아먹는다. "어떤 거?"를 다섯 번 되묻는 동안.

**해결책(다음에 시도):**
- **이름을 화면에 박아둔다.** 와이어프레임/피그마 각 화면·컴포넌트에 `위험감지 바텀시트`처럼 **공식 명칭을 라벨로** 붙이고, 그 이름으로만 부르기로 합의.
- **용어집(Glossary) 한 장.** "위험도 단계 = 안전/의심/경고/위험", "보이스피싱 유형 enum = 기관사칭/계좌이체/납치협박/원격앱". 코드 enum과 한국어를 1:1로 못 박는다. (이번에 카톡에서 `phishingType` enum 한국어 매핑을 뒤늦게 물어본 게 딱 이거다.)
- 말로 설명할 때 **화면을 띄워놓고 가리키기.** 추상적인 지칭 대신 "이 화면, 이 버튼". 손가락이 제일 정확한 명세다.

근본적으로는 **팀의 공용 어휘(ubiquitous language)** 가 없었다. 이름을 먼저 합의했어야 했다.

## 5. 이번 프로젝트는 왜 다들 재밌었을까

이번엔 진짜 다들 열심히 했고, 분위기도 좋았고, 순조롭게 끝났다. 왜였을까를 일부러 분석해본다 — **재미는 운이 아니라 재현 가능한 조건**이라고 믿기 때문에.

- **주제가 와닿았다.** 보이스피싱은 누구나 가족 얘기로 치환된다. "이게 진짜 누군가를 구할 수도 있다"는 감각이 있으면 사람은 더 한다.
- **"되는 걸" 일찍 봤다.** 초반에 BlackHole로 통화 오디오 분리 PoC가 실제로 돌아간 순간, "이거 된다"가 팀 전체의 연료가 됐다. 막연한 기획이 아니라 손에 잡히는 데모가 일찍 있었다.
- **역할이 안 겹쳤다.** 프론트/백/AI가 명확히 나뉘어서 각자 자기 영역의 주인이었다. 주인의식이 있으면 재밌다.
- **막힘이 적었다.** Swagger 계약 덕에 서로 기다리는 시간이 적었고, mock 토글로 남 끝나길 안 기다리고 병렬로 갔다. **블로킹이 적으면 몰입이 끊기지 않는다.**
- **농담이 살아있었다.** 카톡 보면 절반이 드립이다. 긴장 속에서도 웃을 여유가 있었다는 거고, 그게 번아웃을 막았다.

> 결론: 재미는 **① 의미 있는 주제 ② 일찍 보는 성공 ③ 명확한 역할 ④ 낮은 블로킹 ⑤ 농담의 여유** 에서 나왔다. 다음에도 이 다섯 개를 의도적으로 세팅하자.

## 6. 본선 팀과의 차이 — "지금 당장 적용 가능한가"

본선 올라간 팀들을 보니 솔직히 **개발은 우리가 더 많이 했다.** 기술 스택도, 구현 깊이도 우리가 위였다고 느꼈다. 그런데 그들과 우리의 차이는 딱 하나였다 — **"지금 당장 적용 가능한 정도".**

우리는 "기술적으로 되는 것"을 증명하는 데 힘을 썼고, 그들은 "이게 내일 현장에 깔린다면?"을 보여줬다. 심사위원은 부장~본부장, 카카오 리더급이다. 그들에게 꽂히는 건 코드의 깊이가 아니라 **현실에 박히는 그림**이었다.

이건 카톡 사전 QnA에서 멘토가 했던 말이랑 정확히 같다 — *"너무 기술로 가스라이팅하지 마라."* 그때 흘려들었는데, 본선 결과가 그걸 증명했다.

**다음에 새길 것:**
- 발표의 무게중심을 "우리가 이렇게 어려운 걸 했다"에서 **"이게 누구에게, 언제, 어떻게 쓰인다"** 로 옮긴다.
- 기술 깊이는 **신뢰의 근거**로만 쓰고(되니까 믿어라), 스토리는 **적용 가능성**으로 끌고 간다.
- "이게 실제로 깔리려면 뭐가 더 필요한가"를 한 슬라이드라도 정직하게. 그게 오히려 현실감을 준다.
- 개발을 덜 하라는 게 아니라, **개발한 만큼을 "적용 가능성"의 언어로 번역**하는 데 시간을 더 썼어야 했다.
